Summary

Vulcan and loom solve overlapping but distinct problems. Vulcan is an LLM-powered data + ES|QL generator built for rapid Agent Builder prototyping from a single prompt. Loom is a structured SA pipeline that converts customer discovery through qualification, scripting, data modeling, and full-stack deployment into a persistent demo environment. They are most powerful used together, not as alternatives.

Adopt from Vulcan — 4 areas

Capabilities where Vulcan is materially ahead and adoption fills real gaps in loom.

ES|QL generation & testing Adopt from Vulcan
Vulcan
First-class. QueryStrategyGenerator, parameterized ES|QL, RAG pipelines, query_test_runner validates against cluster, compliance checker auto-repairs.
Loom
ES|QL appears in demo scripts and bootstrap.py but is hand-authored per engagement — no automated generation or validation loop.
Recommendation
High priority. Adopt Vulcan’s ES|QL generation + test runner as input to weave-model. Validated parameterized queries replace hand-authored ones in bootstrap.py.
Integration grounding (EPR) Adopt from Vulcan
Vulcan
Fetches Elastic Package Registry, aligns field names to integration schemas, auto-names indices to logs-*/metrics-* for integration dashboard compatibility.
Loom
No EPR integration. Index names and fields manually specified from the demo script.
Recommendation
Medium priority. When a demo involves Fleet/Beats integrations, Vulcan’s EPR grounding ensures generated data is compatible with shipped integration dashboards.
Eval / test infrastructure Adopt from Vulcan
Vulcan
Formal eval framework: Layer 1 (deterministic validation) + Layer 2 (execution metrics). pytest suite, HTML eval reports, query_testing_results.json per demo.
Loom
evals.json per skill (prompt + expected_output + assertions). No automated execution framework — evals are manual/LLM-reviewed.
Recommendation
Medium priority. Adopt Vulcan’s two-layer eval runner pattern for automated skill regression testing — especially weave-model and warp-listen.
RAG query generation Adopt from Vulcan
Vulcan
Explicit MATCH → RERANK → COMPLETION pipeline generation. INFERENCE_RERANK_ENDPOINT + INFERENCE_COMPLETION_ENDPOINT. Parameterized RAG queries with real sample values.
Loom
RAG covered in script + data model (semantic_text, EIS endpoints). No automated RAG query generation or cluster validation.
Recommendation
Medium priority. Vulcan’s parameterized RAG query generation with cluster-validated outputs would improve weave-model for semantic/search demos.

Complementary — 3 areas

Where each tool solves a different part of the same problem. Wiring them together multiplies output quality.

Loom
Agent skill pipeline. Discovery notes → structured JSON → bootstrap.py via SKILL.md instructions.
Integration path
Different layers. Vulcan generates the data layer bottom-up from a prompt; loom builds top-down from customer discovery. Most powerful together.

Loom
weave-model produces explicit JSON mappings + seed data spec. Hot-only ILM default. EIS for inference. demo_critical_docs pattern.
Integration path
Vulcan’s LLM-driven synthetic data gen complements loom’s explicit mapping contracts. Wire Vulcan’s data_generator.py output into the data-model spec as an optional fast path.
Agent Builder Complementary
Vulcan

Loom
First-class via weave-agent skill + deploy step 13. D-029 covers skill_ids, tool types, probe detection. weave-cost (D-036) adds AI cost dashboard.
Integration path
Vulcan focuses on ES|QL tool generation; loom covers the full Agent Builder lifecycle (design → spec → deploy → teardown → cost visibility).

Proposed integration architecture

How Vulcan and loom compose in an SA workflow.

Stage A — Discovery
loom only
warp-listen → thread-qualify (MEDDPIC gate) → thread-audit. Vulcan has no equivalent and does not need one.
Stage B — Data + Queries
Vulcan → loom
Vulcan generates synthetic datasets + validated ES|QL queries grounded on integrations. Output (CSV + all_queries.json) feeds weave-model, replacing hand-authored seed data and queries.
Stage C — Deploy + Ops
loom only
bolt-launch generates bootstrap.py covering spaces, dashboards, ML, SIEM, SLOs, workflows, Agent Builder, weave-cost. Teardown + manifest. Vulcan has no equivalent.

Fundamental design differences

Vulcan

  • Prompt → LLM code generation (Python modules)
  • Bottom-up: data first, queries second
  • Temporary indices — deleted after validation
  • Streamlit UI — not agent-instruction-driven
  • LLM manages the generation loop
  • Single-session, no engagement persistence
  • Agent Builder + ES|QL as the primary deliverable

Loom

  • Discovery notes → structured artifacts (SKILL.md stages)
  • Top-down: customer need → script → data model
  • Persistent engagement environment (bootstrap.py + manifest)
  • Agent-instruction driven — Cursor / Claude Code executes
  • SA reviews and approves before any cluster spend
  • Multi-session engagement workspace per customer
  • Full-stack Kibana: dashboards, ML, SIEM, SLOs, workflows
