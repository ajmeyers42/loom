What changed — item by item

Each recommendation from the initial comparison and its current status.

ES|QL generation & testing Adopt ← before → Integrated
Before
ES|QL was hand-authored per engagement in bootstrap.py with no automated generation or validation loop. Vulcan had a full QueryStrategyGenerator + cluster-based test runner; loom had neither.
After
New weave-query skill (Stage 4.5) extracts ES|QL context from the demo script, invokes Vulcan’s query strategy generator, collects all_queries.json and query_testing_results.json, and writes {slug}-vulcan-queries.json as a typed contract consumed by weave-model.
Delivery: skills/weave-query/SKILL.md · skills/loom/SKILL.md Stage 4.5 · skills/weave-model/SKILL.md Step 0b · commit 1ab9f56
Integration grounding (EPR) Adopt ← before → Integrated
Before
No EPR integration. Index names and fields were manually specified from the demo script, with no guarantee of compatibility with shipped Fleet/Beats integration dashboards.
After
weave-query accepts use_integrations: true and confirmed_integrations list. Vulcan fetches EPR schemas, aligns field names to integration conventions, and generates logs-*/metrics-* index names. The integration_grounded flag in {slug}-vulcan-queries.json prevents downstream renaming that would break dashboard compatibility.
Delivery: skills/weave-query/SKILL.md Step 1 integration grounding section · skills/weave-model/SKILL.md Step 0b integration-grounded note · commit 1ab9f56
RAG query generation Adopt ← before → Integrated
Before
RAG pipelines (MATCH → RERANK → COMPLETION) were covered in demo scripts and data model specs but without automated generation or cluster-validated parameter values.
After
weave-query sets include_rag: true when semantic/RAG scenes are detected in the script. Vulcan generates the full MATCH → RERANK → COMPLETION pipeline with real sample_values from the indexed data. The rag_pipeline block in {slug}-vulcan-queries.json drives semantic_text field placement and inference endpoint wiring in weave-model.
Delivery: skills/weave-query/SKILL.md Step 1 + reference table · {slug}-vulcan-queries.json schema with rag_pipeline field · commit 1ab9f56

Summary

Vulcan and loom solve overlapping but distinct problems. Vulcan is an LLM-powered data + ES|QL generator oriented around rapid Agent Builder prototyping from a single prompt. Loom is a structured SA pipeline that converts customer discovery through qualification, scripting, data modeling, and full-stack deployment into a persistent demo environment. They are most powerful used together, not as alternatives.

Adopt from Vulcan — 4 areas

Capabilities where Vulcan was materially ahead and adoption would fill real gaps.

ES|QL generation & testingAdopt from Vulcan
Vulcan
First-class. QueryStrategyGenerator, parameterized ES|QL, RAG pipelines, query_test_runner validates against cluster, compliance checker auto-repairs.
Loom
ES|QL appears in demo scripts and bootstrap.py but is hand-authored per engagement — no automated generation or validation loop.
Recommendation
High priority. Adopt Vulcan’s ES|QL generation + test runner as input to weave-model.
Integration grounding (EPR)Adopt from Vulcan
Vulcan
Fetches Elastic Package Registry, aligns field names to integration schemas, auto-names indices to logs-*/metrics-* for integration dashboard compatibility.
Loom
No EPR integration. Index names and fields manually specified from the demo script.
Recommendation
Medium priority. When a demo involves Fleet/Beats integrations, Vulcan’s EPR grounding ensures generated data is compatible with shipped integration dashboards.
Eval / test infrastructureAdopt from Vulcan
Vulcan
Formal eval framework: Layer 1 (deterministic validation) + Layer 2 (execution metrics). pytest suite, HTML eval reports, query_testing_results.json per demo.
Loom
evals.json per skill (prompt + expected_output + assertions). No automated execution framework — evals are manual/LLM-reviewed.
Recommendation
Medium priority. Adopt Vulcan’s two-layer eval runner pattern for automated skill regression testing.
RAG query generationAdopt from Vulcan
Vulcan
Explicit MATCH → RERANK → COMPLETION pipeline generation. Parameterized RAG queries with real sample values from indexed data.
Loom
RAG covered in script + data model (semantic_text, EIS endpoints). No automated RAG query generation or cluster validation.
Recommendation
Medium priority. Vulcan’s parameterized RAG query generation with cluster-validated outputs would improve weave-model for semantic/search demos.

Complementary — 3 areas

Where each tool solved a different part of the same problem. Integration paths were open questions.

Streamlit app + LLM Python codegen. Prompt → data_generator.py + query_generator.py modules.
Loom
Agent skill pipeline. Discovery notes → structured JSON → bootstrap.py via SKILL.md instructions.
Integration path
Different layers. Vulcan generates the data layer bottom-up from a prompt; loom builds top-down from customer discovery. Bridge was not yet defined.
Data modeling & generationComplementary
Vulcan
LLM-generated pandas DataFrames → CSV → bulk index. Infers mappings from dtypes. semantic_text + inference id. TSDS/lookup/data_stream modes.
Loom
weave-model produces explicit JSON mappings + seed data spec. Hot-only ILM default. EIS for inference.
Integration path
Consider wiring Vulcan’s data_generator.py output into loom’s data-model spec as an optional input path. (Aspirational — not yet implemented.)
Agent BuilderComplementary
Vulcan
Primary focus. ES|QL tool generation + deploy via /api/agent_builder.
Loom
First-class via weave-agent + deploy step 13. Full lifecycle: design → spec → deploy → teardown → cost visibility.
Integration path
Vulcan and loom target the same Agent Builder API from different angles but shared no artifacts.
